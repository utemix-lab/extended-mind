# Object Types — Extended-mind v0.1

Этот документ определяет **типы сущностей (object types)**,  
с которыми работает extended-mind как система мышления (Мир-3).

Он **не делит объекты по репозиториям**  
и **не жёстко привязывает их к проектам**.

Объект определяется как:

> **Тип + Контекст(ы) принадлежности**

---

## 1. Принципиальная модель

### 1.1. Тип ≠ Проект

- Тип отвечает на вопрос: **“что это?”**
- Контекст отвечает на вопрос: **“где и зачем это используется?”**

Один и тот же тип объекта может:
- существовать в нескольких проектах,
- иметь разные роли,
- быть проиндексированным или нет — в зависимости от контекста.

---

### 1.2. Контексты принадлежности (не исчерпывающий список)

- `extended-mind` — базовый, родительский контекст
- `vova-and-petrova`
- `thin-tank`
- `evoquant`
- `word-machine`
- `author-line`
- `machine-log`

Контекст задаётся:
- тегами,
- метаданными,
- размещением в структуре репозитория.

---

## 2. Базовые типы объектов (core)

### 2.1. `definition`
**Что это:** каноническое определение понятия.  
**Роль:** стабилизация терминологии.

- индексируется: да
- изменяется редко
- может использоваться всеми проектами

---

### 2.2. `principle`
**Что это:** базовый принцип или ценность системы.  
**Роль:** ограничитель спекуляций и дизайнерских соблазнов.

- индексируется: да
- выше паттернов, ниже ADR

---

### 2.3. `manifest`
**Что это:** декларативный документ уровня “зачем мы это делаем”.  
**Роль:** смысловая рамка.

- индексируется: да
- допускает более литературный стиль

---

## 3. Архитектурные типы

### 3.1. `adr`
**Что это:** зафиксированное архитектурное решение.  
**Роль:** высший авторитет при конфликтах.

- индексируется: да
- изменяется только новым ADR

---

### 3.2. `pattern`
**Что это:** воспроизводимый способ действия.  
**Роль:** “как делать”, не “почему”.

- индексируется: да
- может быть:
  - глобальным (extended-mind),
  - локальным (thin-tank / V&P).

---

### 3.3. `concept-map`
**Что это:** обзорная карта связей.  
**Роль:** навигация по системе и проектам.

- индексируется: да
- не содержит детальной аргументации

---

## 4. Проектные типы

### 4.1. `project`
**Что это:** каркас проекта.  
**Роль:** зафиксировать:
- цель,
- границы,
- уровень зрелости,
- связи с другими проектами.

- индексируется: да (в виде project-frame)
- не включает сырые гипотезы

---

### 4.2. `project-status` (опционально)
**Что это:** текущий статус проекта.  
**Роль:** дать extended-mind понимание зрелости.

- индексируется: опционально
- полезно для навигации и планирования

---

## 5. Story-layer типы (пограничные)

### 5.1. `story-node`
**Что это:** фрагмент художественного/рефлексивного текста.  
**Роль:** сырьё для:
- сценариев,
- author-line,
- будущей структуризации.

- индексируется: **нет (v0.1)**
- может быть преобразован в другие типы

---

### 5.2. `author-line`
**Что это:** связанная повествовательная линия автора.  
**Роль:** художественный и перформативный слой.

- индексируется: нет
- используется для контента, не для ответов

---

### 5.3. `machine-log`
**Что это:** машинно-ориентированная хроника процесса.  
**Роль:** фиксация хода разработки.

- индексируется: нет (v0.1)
- может служить источником story-node

---

## 6. Технические и служебные типы

### 6.1. `spec`
**Что это:** спецификация поведения системы.  
**Роль:** операционные правила extended-mind.

- индексируется: да
- основа RAG Core

---

### 6.2. `core`
**Что это:** документы ядра системы.  
**Роль:** сборка архитектуры как целого.

- индексируется: да
- изменяется осторожно

---

## 7. Связь типов с RAG

| Тип           | Индексируется v0.1 | Комментарий |
|---------------|--------------------|-------------|
| definition    | да                 | канон |
| principle     | да                 | ограничитель |
| manifest      | да                 | рамка |
| adr           | да                 | высший приоритет |
| pattern       | да                 | рабочие методы |
| project       | да (frame)         | каркас |
| concept-map   | да                 | навигация |
| spec / core   | да                 | поведение системы |
| story-node    | нет                | художественное сырьё |
| author-line   | нет                | перформанс |
| machine-log   | нет                | поток |

---

## 8. Почему это важно

Этот документ позволяет:

- не путать проекты и типы,
- не перегружать RAG,
- безопасно добавлять новые проекты,
- в будущем легко перейти к графу (тип → связь → тип).

---

## 9. Статус

Object Types v0.1 фиксирует онтологию extended-mind.  
Все новые документы должны явно указывать свой тип и контекст(ы).
