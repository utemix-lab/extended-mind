# Character / Workbench — Spec v1

Баланс архитектурного и нарративного слоёв: механика фиксируется, сценография остаётся свободной.

## 1) Зачем нужен Character / Workbench

Система одновременно должна быть навигацией, контент‑фабрикой, портфолио и лабораторией.  
Character / Workbench локализует сложность и не допускает онтологического раздувания.

## 2) Базовые определения

**Character** — персонифицированная точка зрения на систему, носитель практик и публичная проекция.  
**Workbench** — локальная архитектура работы персонажа (процесс, а не объект).

Character не равен домену/профессии/категории/пользователю.  
Workbench не равен проекту/разделу/узлу канона.

## 2.1) Типы Workbench’ей (v1)

1. **Доменные (контентные)**  
живут внутри доменов (Вова, Анимация +)

2. **Метасистемные**  
работают над системой (Brand Direction, Circus Technologies)

3. **Коллаборативные / прикладные**  
обслуживают систему как продукт (As if)

## 2.2) Нейминг (системный vs нарративный слой)

- Системный слой использует **английский канонический нейминг** (IDs, заголовки workbench’ей).
- Нарративный слой допускает **двуязычные названия** и игровые формы.
- Нейминг как фича бренда закреплён за workbench’ом **Brand Direction** (Petrova).

## 2.3) System Presences (нарративное выделение)

- Нарратив может выделять “присутствия” (например, AI как голос системы).
- В каноне это остаётся **character** с теми же правилами.

## 2.4) Projection of Capability

- Активы, портфолио и практики не являются владением персонажа.
- Персонаж — **носитель демонстрации**, а не “владелец”.
- Это защищает систему от онтологии “всё обо всём”.

**See:** `SYSTEM_LANGUAGE_LAYER.md`, `PRACTICES_SPEC.md`

## 3) Слои Character (5)

1. **Фокус** — что персонаж замечает в мире  
2. **Практики** — как он действует  
3. **Проекции** — где он проявляется (внутр./внешн.)  
4. **Workbench’и** — способы организации работы  
5. **Визуализация** — технологический образ и характерные черты

Один персонаж может иметь несколько workbench’ей.

## 4) Связь с мета‑целями

| Мета‑цель | Роль Character / Workbench |
|---|---|
| Навигация | человеческий вход вместо дерева категорий |
| Контент | источник дайджестов и следов |
| Портфолио | следы работы вместо списков |
| Лаборатория | автономный эксперимент |
| Самоописание | story‑nodes через перспективу |
| Эмерджентность | новые персонажи = новые способы быть системой |

## 5) Архитектурные ограничения

**Запреты**
- Character не владеет доменами
- Character не заменяет онтологию
- Workbench не становится узлом канона
- Character не превращается в пользователя

**Защитные принципы**
- Character проводит, а не “объясняет всё”
- Workbench локален, не универсален
- Рост выносится в каталоги/проекции
- Канон остаётся минимальным

**Ограничение v1**
- не более 2–3 workbench’ей на персонажа

## 6) Где это живёт

**Канон (`universe.json`)**
- только Character как узел
- связи: `relates → domains`, `relates → practices`
- Workbench не живёт в каноне

**Документы**
- `CHARACTERS.md` — кто есть кто и зачем
- `WORKBENCHES.md` — типы workbench’ей (без конкретики)
- `docs/workbenches/WORKBENCH_INDEX.md` — минимальный реестр workbench’ей

**Данные / проекции**
- Story‑nodes — фиксация шагов и находок
- Catalog items — следы работы (сервисы/модели/сигналы)
- Query Mode — основной инструмент навигации

## 7) Preset Context (обязательное поле)

Workbench всегда содержит **Preset Context** — именованный снимок Query Mode:

- набор `projections`
- опциональные `default_refines`
- опциональные `output_actions`

Это декларация намерения для будущей UI‑реализации, не режим и не узел.

## 8) UI‑модель

Пользователь видит:
- персонажа как проводника
- Story как сцену
- Service как действие
- Query Mode как результат

Короткая формула:
**Character** — “с кем я иду”, **Practice** — “как мы идём”, **Query Mode** — “что можно сделать сейчас”.

## 9) Пример (эскиз)

**Петрова**  
Фокус: визуальная культура, анимация, брендинг  
Практики: режиссура, визуальная навигация  
Проекции: story‑сцены, внешние площадки  
Workbench’и: Анимация +, Brand Direction, Circus Technologies

Ни один workbench не требует узла в каноне.

## 10) Почему это не ломает систему

- онтология остаётся минимальной  
- навигация идёт через проекции  
- контент не становится источником истины  
- персонажи масштабируются без переписывания архитектуры

## 11) Workbench readiness (v1)

- Чёткий фокус (1–2 домена)
- Ясные входные сигналы
- Конкретные выходы (catalog/story/content)
- Ограничения (non_goals)
- Preset Context определён

## 12) Что делать дальше (без спешки)

- описать 1–2 типа workbench’ей (без реализации)
- связать первые story‑nodes с персонажами
- поддерживать реестр workbench’ей и проверять readiness
