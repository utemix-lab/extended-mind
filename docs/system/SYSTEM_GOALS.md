# SYSTEM GOALS

Каждая цель описывает:
- формулировку
- почему важна
- признаки прогресса (не KPI)

## Типы целей (Goal types)
- **UG (User Goals)** — что человек может сделать/понять лучше
- **LG (Lab Goals)** — что система и агенты способны производить/проверять/эволюционировать
- **MG (Media Goals)** — как система генерирует доверяемый пред‑контент
- **SG (System Goals)** — свойства, которые архитектура должна сохранять

**Rule:** no idea exists without at least one **UG** or **LG**.  
MG/SG can amplify, but not justify alone.

## User goals (UG)
### [UG-1] Быстро найти “то, о чём слышал”
**Почему важно:** снижает порог входа и превращает систему в карту возможностей.  
**Признаки прогресса:** пользователь видит 3–5 релевантных вариантов после 1–2 действий.

### [UG-2] Увидеть связанный контекст, а не список
**Почему важно:** ценность в маршруте мышления, а не в каталоге.  
**Признаки прогресса:** пользователь может сохранить/повторить путь (через теги/узлы/практики).

### [UG-3] Двигаться по системе как по карте, а не каталогу
**Почему важно:** Query Mode должен быть ощущаемым, но не ломать навигацию.  
**Признаки прогресса:** теги включают режим, но граф остаётся стабильным.

### [UG-4] Получать обновляемую картину мира
**Почему важно:** система не должна выглядеть “замороженной”.  
**Признаки прогресса:** видно, что появилось новое и куда идти дальше.

## Lab goals (LG)
### [LG-1] Система как живая лаборатория мышления
**Почему важно:** система должна позволять тестировать гипотезы и паттерны.  
**Признаки прогресса:** новые подходы проверяются без перелома канона.

### [LG-2] Минимальный канон + максимальный охват
**Почему важно:** канон — ядро смысла, не база данных.  
**Признаки прогресса:** новые каталоги подключаются без расширения `universe.json`.

### [LG-3] Машины работают по направлению, а не по задачам
**Почему важно:** агенты должны сверять решения с целями и guardrails.  
**Признаки прогресса:** `SYSTEM_CHECKS.md` используется перед изменениями.

### [LG-4] Персонажи как активные агенты знаний
**Почему важно:** персонажи создают траектории и дайджесты.  
**Признаки прогресса:** есть устойчивые дайджест‑сигналы от персонажей.

### [LG-5] Самоописательность системы
**Почему важно:** архитектурные изменения должны оставлять нарративный след.  
**Признаки прогресса:** каждое значимое изменение имеет story‑node и краткое описание для разных аудиторий.

## Media goals (MG)
### [MG-1] Система сама производит пред‑контент
**Почему важно:** ценность — в историях находок, а не в статике.  
**Признаки прогресса:** Query Mode/Practices дают основу для “как мы нашли X”.

### [MG-2] Прозрачность процесса как ценность
**Почему важно:** система должна показывать работу, а не обещания.  
**Признаки прогресса:** внешние ссылки/референсы встроены в поток.

### [MG-3] Персонажи как инфлюенсеры с активом
**Почему важно:** истории должны жить отдельно от каталога.  
**Признаки прогресса:** story‑nodes развиваются без роста канона.

### [MG-4] Самоописание для разных аудиторий
**Почему важно:** система должна объяснять себя потребителю, инвестору, партнёру.  
**Признаки прогресса:** есть краткие форматы (1‑параграф, слайды, туториал) без переписывания смысла.

## System goals (SG)
### [SG-1] Система не ломается от роста данных
**Почему важно:** рост каталогов не должен ломать архитектуру.  
**Признаки прогресса:** новые каталоги подключаются без структурных правок.

### [SG-2] Разделение: entity / context / interaction
**Почему важно:** система всегда знает, что есть что.  
**Признаки прогресса:** нет смешения узлов, проекций и режимов.

### [SG-3] Возможность смены рендера без потери смысла
**Почему важно:** формы могут меняться, смысл — нет.  
**Признаки прогресса:** Dream‑graph/Godot/site показывают одно ядро.
