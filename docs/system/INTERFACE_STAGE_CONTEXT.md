# Interface Stage Context (v0)

Цель документа: дать внешним LLM **компактный контекст** текущего этапа интерфейса, чтобы они не теряли замысел и не “утаскивали” систему в обычный веб.

## 1) Что сейчас происходит
- Мы строим **видимую механику** интерфейса, а не добавляем новые сущности.
- Главная задача — сделать **архитектуру читаемой в UI** (контексты, ответы, роли).
- Интерфейс в режиме Visitor остаётся read‑only; канон не меняется из UI.

## 2) Ядро механики (коротко)
- **Actors / Instruments / Matter** — язык ролей сцены.
  - Actor: на кого смотрим (узлы канона).
  - Instrument: что делаем (режимы/действия/пресеты).
  - Matter: что отвечает мир (каталоги/выдача).
- Query Mode — **реакция на контекст**, не отдельная “страница”.
- Контекст формируется кликами по маркерам (`cap:`, `provider:`, `domain:`).

## 3) Текущие UI‑якоря (в Visitor)
- **Context strip**: показывает активный контекст (owner/projection/entity) и тип ответа.
- **Response type**: Results / Actions / Steps / Info — видимый индикатор.
- **Виджеты**: визуальные маркеры, которые запускают реакцию (hover/контекст).

## 4) Что НЕ делаем на этом этапе
- Не добавляем новые узлы‑каталоги.
- Не строим фасет‑океаны и сложные фильтры.
- Не вводим RAG/LLM.
- Не превращаем Query Mode в “страницу”.

## 5) Смысл этапа
Сделать так, чтобы пользователь **ощущал**, почему интерфейс отвечает именно так,  
а команда видела “внутренние рычаги” без чтения кода.

## 6) Мини‑пакет для внешних LLM
Рекомендуемый комплект из 3 файлов:
- `SYSTEM_OVERVIEW.md`
- `PROJECT_OVERVIEW.md`
- `extended-mind/docs/system/INTERFACE_STAGE_CONTEXT.md`
