# ADR-005: Механизм Signature Decisions

**Статус:** Принято  
**Дата:** 2026-01-27

---

## Signature

- [x] **self-aware-not-narcissistic** — система знает о себе, но не говорит интерфейсом

---

## Контекст

Система формирует архитектурный стиль через повторяющиеся решения. Нужен механизм для:
1. Отслеживания авторских решений
2. Связи практик с примерами реализации
3. Формирования портфолио персонажей

Вопрос: где и как фиксировать оригинальность?

## Решение

**Оригинальность фиксируется как следы, не как мнение.**

### Механизм

1. **ADR содержат `Signature` секцию** — список маркеров авторского стиля
2. **Маркеры — это паттерны**, а не теги: `practice-first`, `perception-ontology-split`, etc.
3. **Портфолио = запрос по маркерам**, не новые сущности канона

### Маркеры стиля

| Маркер | Значение |
|--------|----------|
| `practice-first` | Приоритет практик над таксономиями |
| `perception-ontology-split` | Разделение восприятия и онтологии |
| `self-aware-not-narcissistic` | Система знает о себе, но не говорит интерфейсом |
| `story-as-evolution-unit` | Story-node как атом эволюции |
| `graph-over-hierarchy` | Граф вместо иерархии |

### Что НЕ делаем

- ❌ Не создаём `signature` как тип узла
- ❌ Не добавляем edges practice → ADR в канон
- ❌ Не вводим "innovation score" или подобное

## Последствия

**Положительные:**
- Портфолио персонажей формируется автоматически через маркеры
- Система может объяснить свой стиль через примеры, а не декларации
- Канон остаётся чистым (знания о мире, не о себе)

**Отрицательные:**
- Требуется consumer-модуль для сборки портфолио по маркерам

## Связь с персонажами

Петрова (брендинг) курирует:
- `practice-branding`
- ADR с маркером `perception-ontology-split`
- Story-nodes про визуальные решения

Её портфолио = `query ADR WHERE signature CONTAINS 'perception-ontology-split'`

---

## Ключевой принцип

> Система создаёт условия, в которых оригинальность оставляет следы.
