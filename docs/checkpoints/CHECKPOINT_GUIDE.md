# Checkpoint Guide

> Практика периодической фиксации состояния системы.

---

## Зачем

1. **Контекст для агентов** — быстрый старт после сбоя/смены
2. **Защита от дрейфа** — отслеживание, не ушла ли система не туда
3. **История решений** — почему сделано так, а не иначе

---

## Когда создавать checkpoint

- После значительных изменений (уборка, новая фича)
- Перед длинной паузой в работе
- При смене агента / контекста
- Раз в неделю при активной работе

---

## Формат

```markdown
# Checkpoint YYYY-MM-DD

## Статус системы

| Компонент | Состояние | Заметки |
|-----------|-----------|---------|
| universe.json | ✅ / ⚠️ / ❌ | краткое описание |
| dream-graph | ... | ... |
| extended-mind | ... | ... |
| contracts | ... | ... |
| godot-sandbox | ... | ... |

## Что сделано с прошлого checkpoint

- пункт 1
- пункт 2

## Проблемы / Долги

- что сломано или отложено

## Следующий шаг

- конкретное действие
```

---

## Правила

1. **Один файл = один checkpoint** — `YYYY-MM-DD.md`
2. **Краткость** — не дублировать Playbook, только дельта
3. **Честность** — фиксировать проблемы, а не только успехи
4. **Следующий шаг** — всегда конкретный, один

## Чек-лист для агента

При начале работы:

1. Прочитать `SINGLE-SOURCE-PLAYBOOK.md` — замысел
2. Прочитать последний checkpoint — текущее состояние
3. Проверить `universe.json` — канон
4. Начать работу
